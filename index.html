TCP 面试知识点复习

TCP基本知识点

TCP提供面向连接的、可靠的端到端全双工字节流服务

  • 面向连接：发送和接收数据前要建立连接
  • 可靠性保证：超时重传、快速重传、数据校验（丢弃错误数据段并不发送确认信号）、 接收端对收到的数据段按照序号排序，丢弃重复数据段。
  • 端到端: 意味着不能广播和多播
  • 字节流：TCP不解释传输的数据类型，对字节流的解释由双方的应用层解释
  • 全双工服务：每一端都可以同时发送和接受

TCP头部

TCP连接建立与断开

TCP三次握手（序号同步）

  1. TCP连接建立初始，客户端发送SYN报文段，通知服务器端即将建立TCP连接，该TCP报文段仅包含TCP首部，不包含数据。首部包含客户端要建立TCP连接的端口号，当前ISN（初始序号）。32位确认序号为空。
  2. 服务器接收到客户端的连接请求后获得了当前客户端的序号，然后发送包含服务器端初始序号的SYN报文段作为应答，此时首部32位确认序号设置为客户端序号+1表示确认， 注意 ： SYN报文将占用一个序号。
  3. 客户端收到服务器应答后确定了此后发送报文段的序号，然后向服务器发送ack应答，设置32位确认序号为服务器ISN+1，表示自己已收到服务器初始序号，若此ACK丢失则造成服务器端超时重发SYN+ACK，而此时若客户端发送数据，怎造成半打开状态，触发服务器返回复位RST字段。注意 ：此时的确认报文段ACK不占用序号。

TCP四次挥手（全双工机制导致四次挥手）

  1. 请求端应用层程序关闭导致TCP向服务器发送FIN报文段，包含当前序号以及确认序号， 注意： FIN报文段将占用一个序号。
  2. 服务器收到客户端的FIN报文段后发送ACK确认此信号，32位确认序号设置为上一个FIN的0序号+1.一旦ACK丢失，发送FIN的发送端就会在一个MSL时间后超时重传FIN，接收端会重发ACK，（除了FIN之外的其他迟到报文都丢弃，因为可能是此端口上一个连接的迟到报文）。而发送通道仍处于打开阶段， 即此时服务器端仍可以发送数据。
  3. 客户端收到服务器返回的ACK确认报文后关闭发送通道（是一种软关闭，仍可以发送ACK响应报文，但不能发送其他报文，即应用程序进程关闭，发送ACK是系统内核进程）。但仍可接受数据。此时若客户端未收到ACK确认报文（超时），客户端将重传FIN报文。服务器端通常此时会发送一个FIN报文段过来，客户端将发送ACK确认此报文已送达并关闭接收通道。
  4. 服务器收到FIN确认ACK后等待2MSL，然后断开发送通道，否则将超时重传，当发送接收全部关闭（软关闭）时本次TCP连接将全部断开。

TCP连续报文段ACK延迟

由于接收端能够根据序号进行排序，确认序号会返回当前已接收的最大报文序号。

累计确认避免重传

在超时发生之前后面的报文段确认到达，即使当前报文段确认丢失也不重传

快速重传

超时重传存在延迟高的问题，通过冗余ACK可以执行快速重传

2MSL等待

当TCP执行一个主动关闭后，并发回最后一个ACK，该连接必须在TIME_WAIT状态等待2倍的MSL时间，以防止最后的ACK丢失。在此期间收到的其他迟到报文均丢弃掉。

TCP半关闭，半打开

  • 半关闭：一端结束它的发送之后还能接收来自另一端的能力，属于正常状态。
  • 半打开：一端结束了连接，而另一端不知道，一个常见的例子就是一端突然断电

平静时间

平静时间是指TCP复位后MSL时间内不允许建立任何连接，这是为了防止主机在上一个2MSL等待时间内重启，导致上一个替身报文到达。但实际中由于TCP重启都会超过MSL，所以基本没有遵守这一规定的实现。

复位报文段 RST

  • 正常关闭：四次挥手发送FIN关闭连接。
  • 异常关闭：发送端发送RST报文到服务器端，服务器端不返回任何响应直接关闭连接并通知应用层连接被对方复位。RST报文包含一个序号和一个确认序号。

TCP流量控制、拥塞控制、慢启动

  • TCP接收端使用滑动通告窗口控制发送端发送数据快慢

    滑动窗口的大小由接收端可用缓冲区大小决定，发送方通过接收方发送的ACK来获得窗口大小。

  • TCP发送端使用拥塞窗口控制一次发送多少报文段。

    通常TCP连接的两端处于不同的子网，连接传输的数据可能要经过多个路由转发以及速率较慢的链路，此时同时发送多个报文段（即使没有超过接收端通告窗口大小）是不合适的，中间路由器可能因缓存溢出而丢弃分组。

    • 慢启动： TCP连接之后发送端初始化拥塞窗口为一个报文段，当收到一个ACK时拥塞窗口扩大为两倍。直到ACK超时拥塞窗口重置，发送端取拥塞窗口和通告窗口的最小值作为发送上限。

  • TCP拥塞避免算法

    慢启动算法是在一个连接上发起数据流的方法，拥塞避免算法是一种处理分组丢失的方法。

    • 分组丢失有两种指示： 1. 发生超时 2. 收到重复的确认
    • 拥塞避免和拥塞窗口是发送端使用的流量控制，通告窗口是接收端使用的流量控制，前者是发送端对网络拥塞的估计，后者与接收方可用缓存大小有关。
    1. 初始化拥塞窗口为1个报文段，拥塞避免上限ssthresh为65525字节。
    2. 启动慢启动，拥塞窗口指数增长，直到发生拥塞（超时或收到重复ACK）设置ssthresh为拥塞窗口的一半，拥塞窗口设置为1个报文段。
    3. 此后拥塞窗口的增长方式取决于是否超过拥塞避免上限ssthresh，当小于ssthresh时使用慢启动指数增长，当等于ssthresh时之后变为线性增长，每次加一。

拥塞避免算法其实就是在发生拥塞时设定一个慢启动增长上限。

ARP

以广播方式请求目的IP的物理地址。

ICMP

Internet控制报文协议，一般被IP协议或更高层使用，被封装在IP数据报内部。用于在IP主机、路由器之间传递控制消息。控制消息是指网络通不通、主机是否可达、路由器是否可用等网络本身的消息。这些控制消息虽然并不传输用户数据，但是对于用户数据的传递起着重要的作用。ICMP报文有两种：差错报告报文和询问报文。

Ping

用来测试网络中主机是否可达。该程序发送ICMP回显请求报文给目的主机，并等待返回ICMP回显应答。

  • 注意： Ping程序不是一个用户级进程，它是由内核完成的。

Ping程序会在ICMP报文中写入发送时间，当接收到回显报文时计算时间差来表示往返时间。
一般第一个往返时间值会比其他的大，这是由于目标IP没有在告诉ARP缓存中。

MTU

MTU: 链路层最大传输单元。它是指由于网络特性以及环境限制，每个网络都有自己的MTU限制。当IP层传下来的数据报大小超过这个限制，那么IP层就要进行分片。

路径MTU: 当不同网络之间通信时，由于MTU的不同，在传输时数据报的分片情况也不同。

  • 路由选路不对称: A到B的路由和B到A的路由不同。

由于路由可能是不对称的，所以发送和接受的分片情况也不一定相同，在发送前路径MTU会通过发现机制确定。

UDP协议

UDP首部包含源端口号和目的端口号，UDP端口号和TCP端口号是相互独立的。

UDP首部包含覆盖首部和数据的校验字段，而IP首部校验字段只覆盖IP首部。UDP校验字段是可选的，TCP是必需的。

知识点1 无状态和持久化连接

  • HTTP是无状态连接:不保留上一次传输的内容，即不对上一次的请求和响应做持久化处理
  • HTTP1.1之后连接默认是持久化连接：即任意一端没有明确表明断开就一直保持TCP的连接状态

* 持久化连接是指TCP的持久化连接，无状态连接是指请求和响应不做持久化处理

HTTP事务的时延

  1. TCP延迟确认造成的时延： 由于TCP发送数据需要接收端返回ACK确认报文，而ACK确认报文往往会很小，TCP延迟确认就会将ACK放到缓冲区，寻找能够捎带这个ACK的同方向数据报文，如果没找到则单独发送这个ACK，而这个过程会造成时延。尤其是HTTP具有双峰特征的请求应答行为更降低了捎带信息的可能。
  2. Nagle算法： 鼓励TCP发送全尺寸的报文段。 为了提高网络利用率，Nagle算法会等待足够多的数据加载到缓冲区，将足够的TCP数据绑定到一起发送出去，除非所有发送的分组都被确认了才允许发送非全尺寸的分组。这会和TCP延迟确认造成冲突，延迟确认想要找到同方向的数据捎带ACK确认报文，而捎带ACK的报文在其他分组的ACK报文没有全部收到或者缓冲区尺寸不够之前不会发送出去。
  3. 2MSL等待造成时延：TCP关闭一条连接时会进行2MSL等待，在2MSL时间内不允许建立相同ip相同端口的连接，这是为了防止迟到的报文影响新的连接。

HTTP解决时延的办法

  1. 并行连接： 串行化的连接可能会很慢，当一个页面中存在多个HTTP事务时需要建立多条连接同时加载资源。
  2. 持久化连接： HTTP1.1之后所有的连接默认都是持久化的，当然服务器并不一定一直保持连接。持久化连接可以消除TCP连接时延和慢启动时延。
  3. 管道化连接： 管道化连接必须保证连接是持久化的。它是指在发送一个事务时不必等待ACK确认即可发送下一个事务分组，这样可以减少网络环回延迟。

get 和 post 区别

  • 区别一：
    get重点在从服务器上获取资源，post重点在向服务器发送数据；
  • 区别二：
    get传输数据是通过URL请求，以field（字段）= value的形式，置于URL后，并用”?”连接，多个请求数据间用”\&”连接，如http://127.0.0.1/Test/login.action?name=admin&password=admin，这个过程用户是可见的；
    post传输数据通过Http的post机制，将字段与对应值封存在请求实体中发送给服务器，这个过程对用户是不可见的；
  • 区别三：
    Get传输的数据量小，因为受URL长度限制，但效率较高；1024k
    Post可以传输大量数据，所以上传文件时只能用Post方式；
  • 区别四：
    get是不安全的，因为URL是可见的，可能会泄露私密信息，如密码等；
    post较get安全性较高；
  • 区别五：
    get方式只能支持ASCII字符，向服务器传的中文字符可能会乱码。
    post支持标准字符集，可以正确传递中文字符。

链接

各个源代码模块单独的被编译，然后将他们组装成一个整体，这个过程称为链接。被链接的各个部分本身就是二进制文件，链接时需要修正所有对符号地址的引用。

  • 静态链接

    • 静态链接最简单的情况就是在编译时和静态库链接在一起成为完整的可执行程序。这里所说的静态库就是对多个目标文件（.o）文件的打包，通常静态链接的包名为lib**.a，静态链接所有被用到的目标文件都会复制到最终生成的可执行目标文件中。这种方式的好处是在运行时，可执行目标文件已经完全装载完毕，只要按指令序执行即可，速度比较快，但缺点也有很多，在讲动态链接时会比较一下。

    • 既然静态链接是对目标文件的打包，这里介绍些打包命令。

    • 首先编译得到test1.o和test2.o两个目标文件，之后通过ar命令将这两个文件打包为.a文件，文件名格式为lib + 静态库名 + .a后缀。在生成可执行文件需要使用到它的时候只需要在编译时加上即可。需要注意的是，使用静态库时加在最后的名字不是libtest.a，而是l + 静态库名。

  • 动态链接

    • 静态链接发生于编译阶段，加载至内存前已经完整，但缺点是如果多个程序都需要使用某个静态库，则该静态库会在每个程序中都拷贝一份，非常浪费内存资源，所以出现了动态链接的方式来解决这个问题。动态链接分为载入时动态链接和运行时动态链接。

    • 动态链接在形式上倒是和静态链接非常相似，首先也是需要打包，打包成动态库，不过文件名格式为lib + 动态库名 + .so后缀。不过动态库的打包不需要使用ar命令，gcc就可以完成，但要注意在编译时要加上-fPIC选项，打包时加上-shared选项。

    • 使用动态链接的用法也和静态链接相同。

    • 如果仅仅像上面的步骤是没有办法正常使用库的，我们可以通过加-Lpath指定搜索库文件的目录（-L.表示当前目录），默认情况下会到环境变量LD_LIBRARY_PATH指定的目录下搜索库文件，默认情况是/usr/lib，我们可以将库文件拷贝到那个目录下再链接。

  • 比较静态连接个动态链接的优缺点：

    • 动态库运行时会先检查内存中是否已经有该库的拷贝，若有则共享拷贝，否则重新加载动态库（C语言的标准库就是动态库）。静态库则是每次在编译阶段都将静态库文件打包进去，当某个库被多次引用到时，内存中会有多份副本，浪费资源。
    • 动态库另一个有点就是更新很容易，当库发生变化时，如果接口没变只需要用新的动态库替换掉就可以了。但是如果是静态库的话就需要重新被编译。
    • 不过静态库也有优点，主要就是静态库一次性完成了所有内容的绑定，运行时就不必再去考虑链接的问题了，执行效率会稍微高一些。

链接过程

符号解析

  • 可重定位目标文件

    对于独立编译的可重定位目标文件，其ELF文件格式包括ELF头（指定文件大小及字节序）、.text（代码段）、.rodata（只读数据区）、.data（已初始化数据区）、.bss（未初始化全局变量）、.symtab（符号表）等，其中链接时最需要关注的就是符号表。每个可重定位目标文件都有一张符号表，它包含该模块定义和引用的符号的信息，简而言之就是我们在每个模块中定义和引用的全局变量（包括定义在本模块的全局变量、静态全局变量和引用自定义在其他模块的全局变量）需要通过一张表来记录，在链接时通过查表将各个独立的目标文件合并成一个完整的可执行文件。

  • 解析符号表

    解析符号引用的目的是将每个引用与可重定位目标文件的符号表中的一个符号定义联系起来。

重定位

  • 合并节

    多个可重定位目标文件中相同的节合并成一个完整的聚合节，比如多个目标文件的.data节合并成可执行文件的.data节。链接器将运行时存储地址赋予每个节，完成这步每条指令和全局变量都有运行时地址了。

  • 重定位符号引用

    这步修改全部代码节和数据节对每个符号的符号引用，使其指向正确的运行时地址。局部变量可以通过进栈、出栈临时分配，但全局变量（”符号”）的位置则是在各个可重定位目标文件中预留好的。通过上一步合并节操作后，指令中所有涉及符号的引用都会通过一定的寻址方式来定位该符号，比如相对寻址、绝对寻址等。

可执行目标文件

  • ELF头部

    描述文件总体格式，并且包括程序的入口点（entry point），也就是程序运行时执行的第一条指令地址。

  • 段头部表

    描述了可执行文件数据段、代码段等各段的大小、虚拟地址、段对齐、执行权限等。实际上通过段头部表描绘了虚拟存储器运行时存储映像，比如每个UNIX程序的代码段总是从虚拟地址Ox0804800开始的。

  • 其他段

    和可重定位目标文件各段基本相同，但完成了多个节的合并和重定位工作。

    加载

  • 克隆

    新程序的执行首先需要通过父进程外壳通过fork得到一个子进程，该子进程除了pid等标识和父进程不同外其他基本均与父进程相同。

  • 重新映射

    当子进程执行execve系统调用时会先清空子进程现有的虚拟存储器段（简而言之就是不再映射到父进程的各个段），之后重新创建子进程虚拟存储器各段和可执行目标文件各段的映射。这个阶段我们可以理解为对复制来的父进程页表进程重写，映射到外存中可执行文件的各个段。

  • 虚页调入

    加载过程并没有实际将磁盘中可执行文件调入内存，所做的工作紧紧是复制父进程页表、清空旧页表、建立新页表映射工作。之后加载器跳转到入口地址_start开始执行程序，接下来的过程需要配合虚拟存储器来完成。CPU获得指令的虚拟地址后，若包含该指令或数据的页尚未调入内存则将其从外存中调入，调入内存后修改页表得到虚拟页号和物理页号的对应关系。之后重新取同一条指令或数据时因该页已经被调入内存，所以通过虚拟地址得到虚拟页号，虚拟页号通过查页表可以得到物理页号，通过物理页号 + 页内偏移得到具体的物理地址，此时可以通过物理地址取得想要的数据。

strlen 和 sizeof 的区别

  • strlen是字符串处理的库函数，接受参数为字符指针且以‘\0’结尾的字符串。结果在运行时确定。
  • sizeof是运算符，不是函数，可接受任何数据的类型和数据，参数不退化，结果在编译时确定。

引用和指针

  • 引用只是别名，内存中不存在单独实体，编译阶段绑定后不允许更改，只有声明没有定义。指针是具体的变量，占有内存空间，定义后可以更改指向。
  • 引用在声明时必须初始化为另一变量，而指针声明和定义可以分开。
  • 不存在指向空值的引用，但存在指向空值的指针。

结构体内存对齐问题

  • 若无特殊说明，按照结构体中size最大的成员对齐。

C++中可执行文件在内存中的存放方式

  • 分为堆区、栈区、未初始化全局变量区、已初始化全局变量，静态变量，常量数据区、代码区

static作用是什么？在C和C++中有什么区别？

  • static用于修饰局部变量、全局变量、函数。被修饰的变量存储在静态区。静态局部变量生存周期相比普通局部变量变长了，直到程序运行结束而非函数调用结束，且只在第一次被调用时定义。
  • 静态全局变量相比普通全局变量作用域变为只限于本文件，修饰函数时和全局变量一样，都是为了限定访问域。
  • C++中的static还可以修饰类成员，被修饰的类成员不属于任何对象，是所有类实例所共有。
  • static的数据记忆性可以满足函数在不同调用期的通信，也可以满足同一个类的多个实例间的通信
  • static变量未初始化时，默认为0。

结构体和类的区别

  • 结构体的默认限定符是public,类的默认访问控制是private。（就这一个区别！！！）
  • C语言中的结构体不能定义函数，只能描述状态，不能描述行为。

malloc和new的区别

  • malloc和free是标准库函数，支持覆盖；new和delete是运算符，支持重载。
  • malloc仅仅分配内存空间，free仅仅释放内存空间，并不具备调用构造函数和析构函数的功能，用malloc分配内存给类对象存在风险。new和delete除了分配回收功能之外，还会调用构造函数和析构函数。
  • malloc和free返回的是void类型的指针（必须进行类型转换），new和delete返回的是具体类型的指针。

宏定义和函数的区别

  • 宏在编译阶段完成替换，之后被替换的文本参与编译，相当于插入代码，运行时不存在函数调用，执行起来更快速；函数调用需要在运行时跳转到具体调用的函数。
  • 宏函数属于在结构中插入代码，没有返回值，函数调用有返回值。
  • 宏函数参数没有类型，不进行类型检查；函数参数具有类型，需要检查类型。
  • 宏函数不加分号

宏定义和const的区别

  • 宏定义在编译阶段之前替换完成，属于文本插入；const发生在编译过程之中。
  • 宏不检查类型；const会检查类型。
  • const需要占有内存空间，宏定义不需要内存分配。

宏定义和typedef的区别

  • 宏主要用于定义常量和书写复杂的内容，typedef主要用于定义类型别名
  • 宏替换在编译之前完成，typedef是编译的一部分
  • 宏不检查类型，typedef检查类型
  • 宏不是语句，后面没有分号，typedef语句后面要加分号标识结束
  • 注意对指针的操作，typedef char p_char 和 #define p_char char 区别巨大。(p_char a,b时对于typedef来说ab都是指针，而对于宏替换来说只有a是指针，b为字符变量。)

内联函数（inline）的作用

内联函数的作用是为了减少系统调用函数的时间开销，在内联函数调用的地方直接将代码嵌入到函数调用点，这个过程发生在编译阶段。所以一般规模较小但调用频繁的函数一般定义或声明为内联函数。

类中成员函数的定义和声明

  • 类的成员函数声明和定义可以分开，但声明一定要在定义之前。
  • 成员函数的定义可以放在类外，加以作用域限定符进行定义。
  • 类内定义的成员函数默认为内联函数，函数入口在类内。
  • 类外定义的成员函数可以使用inline关键字显示定义为内联函数，也可以定义为普通成员函数。普通成员函数在调用时发生函数调用，而内联函数不需要。类外定义的内联函数要与类定义在同一文件中，否则无法将代码拷贝嵌入到函数调用点。

宏和内联函数（inline）的区别

  • 宏发生在编译前，且不做类型检查，而内联函数在编译时做类型检查，并且有返回值
  • 内联函数强调函数特性，能重载
  • 访问类保护成员或私有成员时只能使用内联函数。

条件编译#ifdef， #else， #endif的作用

  • 减少被编译语句，从而减小目标程序的大小
  • 可以通过加#define，并通过#ifdef来判断，将某些具体模块包括进要编译的内容。

指针常量和常量指针

区分一下几种变量：

  • 前两种都表示定义整型常量a，a的值不允许改变。
  • const int *a 表示定义了一个常量指针a，a是一个指向常量的指针，即*a不允许被赋值。
  • int *const a 表示定义了一个指针常量a，a是一个指针的常量，即a的指向不允许改变。

volatile有什么作用？

  • volatile定义的变量值是易变的，每次用到这个变量的值都要重新读取，而不是读寄存器内的备份。
  • 多线程中被几个任务共享的变量需要定义为volatile。

什么是常引用

  • 常引用可以理解为常量指针，形式为const typename & refname = varname。
  • 常引用下，原变量值不会被别名所修改
  • 原变量的值可以通过原名修改
  • 常引用通常用作只读变量别名或是形参传递（减小内存开销）,最常见的就是类的赋值运算符重载时不允许对右值修改。
  • 常引用允许不同类型变量之间绑定，但此时存在一个中间变量作转化，所以原变量值改变不会影响引用的值

区别以下指针类型

  • int *p[10] 为指针数组，p一个长度为10的一维指针数组。
  • int (*p)[10] 为数组指针，p指向一个长度为10的一维整型数组。
  • int *p(int) 表示一个函数p的声明，返回值是整型指针，参数列表为一个int型变量。
  • int (*p)(int) 表示指向函数的指针p指向一个返回值类型是int，参数类型是int的函数。

a和&a有什么区别

  • a是数组名，是数组首元素的地址，+1表示地址值加上一个int类型的大小，*（a+1） = a[1]
  • &a是数组的指针，其类型是 int (*)[10] ,其值+1时，系统会认为是数组首地址加上整个数组的偏移（10个int型变量），*(\&a + 1) = *(a+11)

数组名和指针（指向数组首元素的指针）区别？

  • 二者均可通过增减偏移量来访问数组中的元素。
  • 数组名不是真正意义上的指针，可以理解为常指针，所以数组名没有自增、自减等操作。
  • 当数组名当做形参传递给调用函数后，就失去了原有特性，退化成一般指针，多了自增、自减操作，但sizeof运算符不能再得到原数组的大小了。

野指针是什么？

  • 野指针是指指向垃圾内存的指针，也叫空悬指针，不是指向null。
  • 产生的原因及解决办法
    • 指针变量没有及时初始化 -> 及时初始化指针，要么置空。
    • 指针free或delete之后没有及时置空 -> 释放内存后立即置空

堆栈的区别

  • 申请方式不同

    栈由系统自动分配
    堆由程序员手动分配

  • 申请大小限制不同

    栈顶和栈底是预先设置好的，大小固定，可以通过ulimit -a查看，通过ulimit -s修改。
    堆向高地址扩展，是不连续的内存区域，大小可以灵活调整

  • 申请效率不同

    栈由系统自动分配，速度快，不会有碎片
    堆由程序员分配，速度慢，且会有碎片

delete 和 delete[ ] 的区别？

  • delete只会调用一次析构函数
  • delete [ ]会调用数组中每个元素的析构函数

虚函数和纯虚函数的区别, 普通成员函数和虚函数的声明和定义问题？

  • 虚函数： 虚函数的作用是实现了动态绑定，目的是通过基类类型的指针指向不同对象时产生相应不同的行为。派生类继承时隐式继承了虚函数属性，派生类中继承的虚函数属于函数重写，要求函数名，参数列表，返回值完全相同。
  • 纯虚函数：只相当于声明了一个接口名，含有纯虚函数的类不能被实例化。纯虚函数就是没有定义只有声明的虚函数。
  • 虚函数可以没有定义只有声明，此时为纯虚函数，不能实例化。能实例化的类必须定义已声明的虚函数，但普通成员函数可以只声明不定义，此时不能调用，否则编译链接时会报错（好像声明了也并没有什么卵用）。

覆盖（重写/重构）、重载、隐藏的区别？

  • 覆盖是派生类中重新定义的函数，其函数名、参数列表、返回值类型和父类完全相同。派生类对象调用该函数时会根据对象类型调用相应的函数。
    • 虚函数重写：动态绑定，根据绑定对象的类型调用。
    • 普通成员函数重写：根据调用该函数的对象或指针类型调用。
  • 重载：具有相同函数名，不同参数列表，不关心返回值类型。
  • 隐藏：当父类函数不是虚函数时，继承过程中重新定义了和父类函数名相同的函数，不关心返回值和参数列表，此时父类同名函数被隐藏，无法调用， 即重载和覆盖之后，原父类函数不能调用。

  • 模板函数是一个通用函数，函数的类型和形参不直接指定而用虚拟类型代表，只适用于参数个数相同而类型不同的函数。

this指针是什么？

  • this是类的指针，指向对象的首地址
  • this只能在成员函数中使用，在全局函数。静态成员函数中都不能使用this。
  • this指针只在成员函数中才有定义。不同编译器存储位置不同。

类模板是什么？

  • 用于解决多个功能相同，数据类型不同的类需要重复定义的问题。

构造函数和析构函数的调用时机

  • 全局对象：构造函数在所有函数调用前执行，在主函数执行完毕时调用析构函数
  • 局部自动对象：建立对象时调用构造函数，函数结束时调用析构函数
  • 动态分配的对象：建立对象时调用构造函数，调用释放时调用析构函数
  • 静态局部变量对象： 建立时调用一次构造函数，主函数结束时调用析构函数

类静态成员函数对数据区的访问限制

  • 由于静态成员函数参数列表中没有this指针，也不允许使用this指针，所以从这个角度来讲静态成员函数只能访问静态数据成员无法使用非静态数据成员也无法调用非静态函数。
  • 如果静态成员函数显示定义了指向对象的指针参数或函数内存在该类对象则可以由参数指针和函数内对象访问非静态成员。

过拟合和欠拟合

过拟合：学习器把训练样本自身的一些特点当做所有潜在样本的都会具有的一般性质，导致泛化性能下降。典型的表现就是训练集样本误差在降低，而测试集样本误差却在升高。

欠拟合：学习器对训练样本的一般性质没有学好，典型表现就是训练误差仍然很高。

有多种因素可以导致过拟合，常见的情况就是学习器的学习能力太强大（模型过于复杂），学习到了很多训练集所包含的不太一般的特性。
而欠拟合通常是因为学习能力低下造成的。

过拟合是无法彻底避免的，我们所能做的就是缓解，或者说减小其风险。

Redis的数据安全和性能保障

Redis持久化选项

Redis提供两种持久化方式：

  1. 快照：也叫时间点存储（point-in-time）,redis会将存在于某一时间点的所有数据全部写入到硬盘中。
  2. 只追加文件（AOF转储）：将所有的写命令追加到硬盘中的只追加文件中。

快照转储

redis客户端会在指定时间或指定规则被满足时自动或手动创建快照。在下一次快照创建完毕之前所有上一次创建快照之后的写操作都不会被记录，对于这部分数据敏感的应用程序影使用AOF只追加文件方式做持久化。

AOF转储

根据设置决定同步频率，每秒同步一次AOF文件时的性能和不使用持久化特性时的性能相差无几。这个性能很大程度取决于硬盘的性能。

解决AOF文件过大的问题

AOF文件由于不断追加会变得越来越大，这有可能耗尽所有硬盘空间，而且redis重启之后重新执行的操作时间也会变得非常长。所以减小AOF文件的大小很有必要。

  • 通过发送bgrewriteAOF命令来移除AOF文件中的冗余命令来重写AOF文件： Redis会创建子进程重写AOF文件，这和快照更新的方式一样，也会产生同样的性能问题和内存占用问题。

Redis复制

Redis通过创建数据副本来分摊客户端的读请求。

全量同步

通常发生在从服务器初始化时，具体步骤如下：

  • 从服务器连接主服务器，发送SYNC命令；
  • 主服务器接收到SYNC命名后，开始执行BGSAVE命令生成RDB文件并使用缓冲区记录此后执行的所有写命令；
  • 主服务器BGSAVE执行完后，向所有从服务器发送快照文件，并在发送期间继续记录被执行的写命令；
  • 从服务器收到快照文件后丢弃所有旧数据，载入收到的快照；
  • 主服务器快照发送完毕后开始向从服务器发送缓冲区中的写命令；
  • 从服务器完成对快照的载入，开始接收命令请求，并执行来自主服务器缓冲区的写命令；

增量同步

Redis增量复制是指Slave初始化后开始正常工作时主服务器发生的写操作同步到从服务器的过程。
增量复制的过程主要是主服务器每执行一个写命令就会向从服务器发送相同的写命令，从服务器接收并执行收到的写命令。

Redis主从同步策略

主从刚刚连接的时候，进行全量同步；全同步结束后，进行增量同步。当然，如果有需要，slave 在任何时候都可以发起全量同步。redis 策略是，无论如何，首先会尝试进行增量同步，如不成功，要求从机进行全量同步。

注意点

如果多个Slave断线了，需要重启的时候，因为只要Slave启动，就会发送sync请求和主机全量同步，当多个同时出现的时候，可能会导致Master IO剧增宕机。

验证快照和AOF文件

Redis可以检查AOF中的错误命令，并去除该命令和之后的所有命令，但遗憾的是Redis并不能修复快照的错误，因为快照是压缩的。

更换故障主服务器

A是主服务器，B为A的从服务器，A发生故障时有两种方法：

  • 向B发送Save命令生成快照，将快照发送给C，设置C为主服务器，B为C的从服务器。
  • 升级B为主服务器

Redis简介

Redis是一个远程内存数据库，它是一个速度非常快的非关系型数据库。它可以存储键与5种不同类型的值之间的映射，也可以将内存中的键值对持久化到硬盘上，可以使用复制特性来扩展读性能，还可以使用客户端分片来扩展写性能。

Redis和memcached

两者性能上相差无几，但功能上存在不同之处：

  1. Redis能自动以两种不同的方式将数据写入硬盘
  2. Redis除了能存储普通的字符串键之外，还可以存储其他四种数据结构，而memcached只能存储普通的字符串键。

Redis既可以被用作主数据库，也可以备用作其他数据库的辅助数据库（二级存储）

Redis由于把数据存储在内存中，所以随机写的速度非常快，并且避免了写入不必要的临时数据。

Redis的5种数据结构

  1. String 字符串：字符串、整数或者浮点数 get set del
  2. List 链表：每个节点包含一个字符串 rpush lrange lindex lpop
  3. Set 集合：无序存储，使用散列表保证值单一 sadd sismember smembers srem
  4. hash 散列表：无序键值对 hset hget hgetall hdel
  5. zset 有序集合：散列表键值对，值为分值，按照分值排序 zadd zrange zrangebyscore zrem

字符串

  • incr : 自增1
  • dect : 自减1
  • incrby : 自增一个数
  • decrby : 自减一个数
  • incrbyfloat : 自增一个浮点数
  • append : 追加到值末尾
  • getrange : 获取子串 [start, end]

SQL SERVER的两种索引：聚集索引和非聚集索引

聚集索引

正文本身按照某种排序方式进行排列，这种正文内容本身就是一种按照一定规则排列的目录称为“聚集索引”

每张表只能有一种排序方式，索引每张表只能有一个聚集索引。
SQL SERVER默认使用主键作为聚集索引

非聚集索引

正文排序是正文排序，目录排序是目录排序，目录只是正文的映射。这种目录纯粹是目录，正文纯粹是正文的排序方式称为“非聚集索引”。

进程管理

进程是为了实现操作系统的并发性和共享性

相关概念

  • 操作系统通过进程控制块PCB来控制和管理进程，PCB描述了进程的状态和基本情况
  • PCB、程序段、相关数据段一起构成了进程映像（实体），进程映像是静态的，进程是动态的。
  • 创建和撤销进程实际上就是创建和撤销PCB
  • 进程是程序的一次执行过程，所以进程是动态的，同时它是系统进行资源分配和调度的一个独立单位。

  • 结束

  • 分配相关资源，资源有限，没有空闲资源分配则进程进入阻塞状态等待资源分配
  • 初始化PCB，包括优先级、各种状态信息等。
  • 插入到就绪队列，等待调度

    进程终止

    正常终止和异常终止
  • 检查PCB状态，正在执行则立即停止
  • 终止所有子进程

  • 删除PCB

  • 恢复上下文
